# 技術説明書 — The Starry Night: Infinite

ゴッホ《星月夜》を「最初の1フレーム」とする、無限に飛行可能な絵画宇宙の Three.js 実装。
単一ファイル `index.html` で完結し、外部画像・外部モデルは一切使用しない(すべて手続き的生成)。

## 2. 「最初の1フレーム」= 絵画の完全再現

### 2.1 ホームカメラと絵画空間マッピング

カメラの初期姿勢(HOME)を固定し、**絵画空間 (u, v)**(u: 左→右 0..1、v: 下→上 0..1)から
ワールド座標への写像 `fromUV(u, v, dist)` を定義している。

```js
const HOME = { pos:(0,10,80), look:(0,26,0) };   // fov 55°
fromUV(u, v, d)  // HOMEカメラの視線基底 + 設計アスペクト1.55の視錐台で
                 // (u,v) 方向のレイを距離 d だけ伸ばした点を返す
```

これにより **絵画上の位置をそのまま指定**して 3D 空間に配置できる。
構図を決める定数はすべてスクリプト冒頭にある:

| 定数 | 内容 |
|---|---|
| `STARS` | 11個の星の `(u, v, サイズ)`。`[3]` が金星(最大) |
| `MOON` | 三日月 `(0.915, 0.835)`、上部右 |
| `VORT_A / VORT_B` | 大渦の主フック(0.40, 0.705)と逆回転カール(0.565, 0.635) |
| `SKY0` | 空の第0シェル中心(距離300) |
| `CYP_BASE / CYP_H` | 糸杉の根元 (-28, 地形高, 30) と高さ 47 |

### 2.2 パララックスによる「平面性の崩壊」

マウス移動(非ドラッグ時)は視線の微回転に加えて **カメラ位置そのものを横/縦に平行移動**させる
(`parPos`、最大 ±約9ユニット)。回転だけでは視差は生じないため、位置オフセットによって
奥行きの異なる筆致レイヤーが互いに滑る=「絵が立体である」ことが最初のマウス操作で露見する。

## 3. 筆致システム(GPU インスタンシング)

### 3.1 基本設計

すべての筆致は **1枚の `InstancedBufferGeometry`** で描画される。CPU 側はインスタンスごとに
乱数シード `aSeed`(vec4)しか持たず、**位置・向き・色・動きはすべて頂点シェーダ内で導出**する。
そのためシェルの再シード(後述)はユニフォーム変更だけで完了する。

```js
ribbonGeo(segX, segY, count, extraAttrs)
// PlaneGeometry(1, 2, segX, segY) を共有ベースにした InstancedBufferGeometry
// position.x ∈ [-0.5, 0.5] = 筆致の長さ方向、position.y ∈ [-1, 1] = 幅方向
```

`segY = 2` のとき中央頂点列が存在し、`buildRibbon()` がそれを法線方向に持ち上げて
**インパスト(厚塗り)の稜線**を作る。遠景用は `1×1`(平板クアッド)に簡略化。

### 3.2 `buildRibbon()`(共通 GLSL)

```glsl
vec3 buildRibbon(world, d0, d1, len, wid, ridge)
```

- 長さ方向の向きは `mix(d0, d1, x)` — 両端でフロー場をサンプルした方向を補間するため、
  **筆致自体が流れの曲率に沿って曲がる**
- 幅は `sin(πx)^0.4` でテーパー(両端が細い)
- カメラ方向と接線の外積で side / normal 基底を構築(常に見えやすい向きに開く)
- `ridge` 係数で中央列を法線方向へ隆起 → 稜線ハイライトの土台

### 3.3 共通フラグメントシェーダ(STROKE_FS)= 絵具の質感

すべての筆致マテリアルが共有する。**全て不透明 + `discard`**(ソート問題を根絶):

| 要素 | 実装 |
|---|---|
| ちぎれた縁 | fbm ノイズ閾値で `discard`(`abs(Y) > 0.70 + 0.55·tear`) |
| 剛毛の溝 | 長さ方向に走る fbm 縞で明度変調 |
| 色の大理石模様 | 別周波数の fbm で ±10% 変調 |
| 稜線スペキュラ | 幅方向位置から擬似法線を再構成し、`|Y|≈0.5` 帯にのみ強い鏡面反射 → **カメラが動くと稜線を光が這う** |
| エミッシブ | `vEmis`(星ハロー・窓・月のみ非ゼロ) |
| 大気遠近 | `exp(-d·uFogD)` で深いウルトラマリン `#18274d` へ |

## 4. フローフィールド(空の魂)

`flowVel(p, seed, R)` — シェルローカル座標で速度場を合成:

1. **渦対**: `vort()`(接線方向 × 距離依存の強度カーブ)。
   - seed = 0(絵画シェル): `uVortA`(+10)と `uVortB`(−7.5)= 絵の二重螺旋そのもの
   - seed > 0: シードのハッシュから渦中心・強度を導出(奥のシェルごとに別の渦構造)
2. **星の攪拌**: 11星それぞれの周囲に小渦(炎の上の陽炎のように星がフローを掻き回す)
3. **空の川**: 高さ `uRiverY` を中心とするガウス帯の中を左→右へ蛇行する水平流
4. **カールノイズ**: `curlN()` による細部の乱れ(`uCurlAmp` は flow モードで変化)

### 移流とリサイクル

```glsl
trav = fract(uTime · 0.011 · uFlow · speed + phase);   // 周期 ≈ 90秒
p    = anchor + normalize(flow(anchor)) · (trav − 0.5) · uTravel;
fade = smoothstep(両端);   // → 幅に乗算
```

筆致はフロー方向に約26ユニット運ばれ、**幅がゼロになってから反対端に再出現**する。
アルファフェードではなく幅収縮なので不透明レンダリングと両立し、リサイクルは不可視。
静止フレームは常に絵画として成立し、動きはその平衡の内側で起こる。

### 色 = 流速の関数

`skyColor()` はウルトラマリン→コバルト→明青のランプを基本とし、
**流速が高い場所ほどターコイズ~クリーム(空の川の色)へシフト**する。
つまり淡い川の帯は「速い流れの可視化」であり、構図と運動が構造的に一致する。

## 5. 深度シェルシステム = 無限飛行

### 5.1 `ShellSet` クラス

```js
new ShellSet(n, radius, firstCenter, dirInto, gap)
// uniform: vec4[n] — xyz = シェル中心, w = シード(0 = 絵画レイアウト)
```

- **近景シェル** `shellsNear`: 6個 × 半径170、間隔105 — フル稜線リボン + 星
- **遠景シェル** `shellsFar`: 8個 × 半径360、間隔260 — 簡略スプライト筆致

毎フレーム `update(camPos, camDir)`:
カメラの**後方に落ちた/遠すぎる**シェルを、視線方向の最遠点(+横ジッター)へ移動し
新しいシードを与える。ジオメトリは触らない(位置はシェーダがシードから再導出)。
→ **どの方向へ飛んでも構造が無限に湧き続ける。**

### 5.2 シード 0 の特権

シェル0のシード 0 は「絵画そのもの」を意味する:
- フロー場は `uVortA/uVortB`(絵の渦)を使用
- 星は `uStar0[12]`(11星 + 月のシェルローカル座標)を使用

奥へ飛んでシェル0がリサイクルされると絵画レイアウトは自然に消え、
**リセット(R)で `ShellSet.reset()` が初期状態を復元** → 深部飛行から平面へ帰る第二の驚き。

### 5.3 LOD 戦略

| 距離帯 | メッシュ | ジオメトリ | フロー評価 |
|---|---|---|---|
| 近景 | `skyNear` | セグメント付きリボン(稜線あり) | 3回/頂点(曲率追従) |
| 遠景 | `skyFar` | 1×1 クアッド、長さ2.5倍 | 1回/頂点 |
| 巨大渦 | `skyFar` 内 | `aSeed.w > 0.985` の個体を長さ8倍・幅6倍に | — |

遠景の「巨大渦」は原画サイズ級の筆致構造として深部を漂う。

## 6. 星・月

### 6.1 ハロー筆致(`halos`)

各星につき同心 4 リング(`ring = floor(aSeed.x·4)`)の周回筆致:

- 角速度はリングごとに逓減、星ごとに回転方向が反転
- 半径は `1 + 0.05·sin(t)` で脈動
- 色: 内側から `白金 → クロムイエロー → オーカー → 暗金`、`vEmis` も内側ほど強い
- **フレック(絵具の飛沫)**: `aSeed.w > 0.90` の個体は半径が時間とともに外側へ漂い、
  幅を縮めながら消える(黄色い絵具の欠片が剥がれ落ちる表現)
- 月(インデックス11、シェル0のみ)はオレンジに着色

### 6.2 コア(`cores`、THREE.Points + 加算合成)

- 星: ガウスコア + グロー + 微弱な回折十字、`1.3 Hz` 前後の瞬き
- **月: フラグメントシェーダ内の SDF(円 − オフセット円)で三日月形**を直接描画
- `gl_PointSize` は距離反比例、最大 320px にクランプ

## 7. 大地 — 静かな対位

### 7.1 ハイトフィールド(二重実装)

```
GLSL_HGT (シェーダ用) ⟷ hgtJS (JS用) — 必ず同期を保つこと
```

- 複数周波数の正弦波で転がる丘
- `z < −240` から立ち上がる山脈項(+52 まで)
- 村の中心 `(25, −140)` 周辺半径 25–90 を smoothstep で**平坦化**(家を置くため)

### 7.2 丘の筆致(`ground`)

- 極座標サンプリング(`pow(r, 0.72)` で手前を高密度化)+ 山脈帯(`aSeed.w > 0.86`)
- 向き = **等高線方向**(勾配の直交)± ジッター
- **風**: `sin(uTime·0.55 − x·0.035 − z·0.03)` の回転項 → 再配向の波が丘を渡っていく
- **雲影**: 移動するノイズで明度 0.78–1.0 を変調(空の流れが地上に落とす影)
- 色: 青緑の畑 / 小麦 / 暗部 / 距離に応じ山青へ、山脈は専用の深青2色

### 7.3 村(`buildVillage`)

- 16軒 + 教会を**手置きシード配列**で配置(`hgtJS` の上に載せる)
- 壁・屋根・尖塔は `BoxGeometry` / `ConeGeometry(4分割=角錐)` を頂点カラー付きで
  `mergeGeometries` → **1ドローコール**。専用シェーダで fbm による塗りムラ + 縦筆の溝
- **窓**(インスタンス化クアッド): `vEmis = 1.6 × フリッカー`(二重正弦の揺らぎ、窓ごとに位相差)
- **屋根のダブ**: 家ごとに26個の小筆致を屋根勾配に沿って散布(こけら葺きの点描)
- **煙突の煙**: 煙突ごとに70本のリボンが `fract(time)` で立ち上り、カールノイズで巻きながら
  風下(+x)へ流れて消える

### 7.4 糸杉(`cypress`)

- 高さパラメータ `t` に沿う細身の輪郭 `4.4·(1−t)^0.7 + 0.35`、ノイズと
  `sin(21t)` の**炎のローブ**で輪郭を波打たせる
- 全体のスウェイは `t²` 加重(根元は不動、先端ほど揺れる)
- `t > 0.72` の筆致はカールノイズ変位 + 方向擾乱が増幅 → **先端が舐めるように揺れる**
- 色は黒緑3段 + 月光側(cos θ > 0)だけ僅かに明るい縁

## 8. 下塗りドーム

半径 3000 の `BackSide` 球(毎フレームカメラに追従、`renderOrder = -1`、深度書込なし)。
垂直グラデーション(地平近くの暗青 → 中天のウルトラマリン)に、
**縦に引き伸ばした fbm を2周波数**重ねて下塗りの筆跡を刻む。
筆致の隙間が「黒い虚無」ではなく「塗られたカンヴァスの奥」に見えるのはこのドームによる。
色はフォグ色 `#18274d` と整合させてあり、遠景筆致がドームに溶け込む。

## 9. 操作系

### 9.1 飛行

| 入力 | 動作 |
|---|---|
| マウス移動 | 位置+回転パララックス(冒頭の奥行き開示) |
| ドラッグ | 視線回転(yaw/pitch、ピッチ ±83° 制限) |
| ホイール / W・S | 視線方向への推力(慣性付き、`exp(−1.5·dt)` 減衰) |
| A・D / Q・E | 横移動 / 上下 |
| Shift | 3倍ブースト |
| 無操作 7 秒 | 視線方向へ微速ドリフト(手放し航行) |
| 地面 | ソフトフロア(`hgtJS + 2.2` 以下に沈まない) |

### 9.2 プリセット(キー 1–6 / R)

イージング付き遷移(smoothstep、2.2–2.6 秒、位置 lerp + 四元数 slerp):

1. **絵画の構図**(= R。シェルも初期化され絵が復元される)
2. 村の屋根の間から空を見上げる
3. 糸杉の樹冠を巡る
4. 月の傍ら
5. **深部の空から村を振り返る**(遠景の村 + 渦巻く空の俯瞰構図)
6. 大渦の心臓部

### 9.3 UI(マウス移動で右下にフェードイン、2.6秒で消える)

- **flow**: becalmed(0.35×)/ van gogh(1×)/ storm(2.2× + カールノイズ増)
- **speed**: 飛行速度 0.2–3×
- **quality**: low / med / high(§10)
- **photo**: UI・ビネット・カーソルを消す(P でトグル、Esc で解除)

## 10. パフォーマンス設計

### 10.1 品質プリセット(`QUALITY`)

| | low | med | high |
|---|---|---|---|
| 近景リボン | 14,000 | 34,000 | 62,000 |
| 遠景スプライト | 22,000 | 52,000 | 98,000 |
| ハロー筆致 | 2,600 | 5,200 | 8,400 |
| 地面筆致 | 12,000 | 26,000 | 44,000 |
| 糸杉 | 1,800 | 3,200 | 5,200 |
| リボン分割数 | 5 | 7 | 9 |

品質変更は該当メッシュのみ dispose → 再生成(`rebuild()`)。
**パレット・フロー速度・第1フレーム構図には一切影響しない**(方針: 忠実性は最後まで守る)。

### 10.2 主要な最適化

- ドローコール総数 ≈ 10(全筆致システムが各1ドロー)
- CPU→GPU 転送はユニフォームのみ(シェル vec4 × 14 + 時刻等)。頂点更新ゼロ
- 全筆致が不透明 + `discard` → ソート不要、オーバードロー最小
- 加算合成は星コア(数十ポイント)のみ
- `frustumCulled = false` + 巨大 boundingSphere(位置がシェーダ内導出のため
  CPU カリング不能だが、頂点シェーダは画面外を安価に棄却)
- `devicePixelRatio ≤ 2`

## 12. 既知の制約

- シェーダのコンパイル可否は実行時にしか検証できない(エラー時は DevTools コンソール参照)
- 構図は数値計算で調整したもので、目視チューニングの余地がある(§11 の定数が調整点)
- 地形関数が GLSL / JS の二重実装のため、片方だけの変更は視覚と衝突判定の不一致を生む
- `file://` 実行のため CDN(ネットワーク接続)が必要
